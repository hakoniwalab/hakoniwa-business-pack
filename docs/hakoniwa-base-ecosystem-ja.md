# 箱庭ベースエコシステムガイド

## 1. はじめに

箱庭は、異なるシミュレータ、プログラム、プログラミング言語、実行環境を組み合わせて、一つのシミュレーションシステムを構成するためのエコシステムです。

箱庭では、すべての機能を一つの巨大なシミュレータやランタイムに統合するのではなく、それぞれのコンポーネントの役割を分離し、必要なものを組み合わせてシステムを構成します。

たとえば、あるシステムでは、物理シミュレーションを行うプログラム、ロボットやドローンを制御するプログラム、センサデータを生成するプログラム、シミュレーション結果を可視化するプログラムが、それぞれ異なるプロセスやプログラミング言語で動作することがあります。さらに、大規模なシミュレーションでは、それらが複数のコンピュータに分散して動作することもあります。

箱庭は、こうした異種・分散したコンポーネントを「箱庭アセット」として扱い、それらを組み合わせて一つのシミュレーションシステムを構成します。

そのためのベースエコシステムとして、箱庭は主に次の仕組みを提供します。

- PDU による共通データモデル
- PDU Registry によるデータ型と Converter の生成
- MBody Registry による物理ボディモデルの管理と展開
- PDU Endpoint による通信の抽象化
- PDU Bridge Core による PDU 転送
- PDU RPC による Request / Response 型通信
- Core PRO / Conductor によるシミュレーション時間の協調

本ドキュメントでは、これらのベースエコシステムがどのような役割を持ち、互いにどのような関係にあるのかを説明します。

MuJoCo、Godot、Hakoniwa Drone、Athrill など、箱庭上で利用できる具体的なアセットやシミュレーション環境については、別ドキュメントで説明します。

## 2. 箱庭アセット

箱庭では、シミュレーションシステムを構成する個々のプログラムやシミュレータを「箱庭アセット」として扱います。

箱庭アセットは、必ずしも箱庭専用に作られたプログラムである必要はありません。既存のシミュレータ、制御プログラム、可視化ツールなども、箱庭のインタフェースを介して接続することで、箱庭エコシステムの一部として利用できます。

たとえば、次のようなものを独立した箱庭アセットとして構成できます。

- 物理シミュレータ
- ロボット制御プログラム
- ドローン制御プログラム
- ECU シミュレータ
- 環境シミュレータ
- 可視化アプリケーション

箱庭の基本的な考え方は、これらを一つのランタイムへ統合することではありません。それぞれのアセットを独立させたまま、共通のデータモデル、通信、データ転送、時刻同期の仕組みを使って協調させます。

この疎結合な構造によって、既存のツールを置き換えるのではなく、異なるツールを組み合わせて新しいシミュレーションシステムを構成できます。

## 3. 宣言型によるシステム構成

箱庭のベースエコシステムでは、通信、データ転送、RPC などのシステム構成を、可能な限り実装コードから分離して定義します。

代表例として、PDU Endpoint は接続仕様を JSON で定義し、PDU Bridge Core は転送対象や転送ポリシーを JSON で定義します。PDU RPC も、Request / Response の構成や利用する PDU を設定ファイルとして宣言的に扱います。

この設計により、アプリケーションコードを変更・再コンパイルすることなく、接続先、通信方式、転送経路、転送頻度などのシステム構成を変更しやすくなります。

箱庭では、

- 「何を実行するか」というアプリケーションロジック
- 「どのようにつなぎ、どのようにデータを流すか」というシステム構成

を分離することを重視しています。

この宣言型の構成は、人間がシステム構造を理解しやすくするだけでなく、AI が構成を解析・生成しやすくすることも意図しています。

## 4. PDU - 箱庭アセット間で交換するデータ

箱庭アセット同士が協調するためには、データを交換する必要があります。

箱庭では、その基本単位を PDU（Protocol Data Unit）と呼びます。

PDU には、たとえば次のようなデータを格納できます。

- ロボットの位置・姿勢
- センサ値
- モータ指令
- 制御指令
- シミュレーション状態
- RPC の Request / Response

箱庭において重要なのは、PDU のデータ構造と、その PDU を実際に運ぶ通信方式を分離していることです。

同じ PDU を、共有メモリ、TCP、UDP、WebSocket、Zenoh など、異なる通信方式で交換できます。

つまり、PDU は「何を交換するか」を定義し、通信基盤は「それをどう運ぶか」を担当します。

この分離が、異なるプログラミング言語、プロセス、コンピュータを組み合わせる箱庭エコシステムの基本となります。

## 5. PDU Registry - 共通データモデルを各実行環境へ展開する

PDU Registry は、箱庭で利用する PDU のデータ型と、それを扱うための Converter などを管理・生成する仕組みです。

PDU の元となるデータ構造は、ROS IDL / message definition を起点として定義できます。この共通定義から、各プログラミング言語や実行環境で利用する PDU データ型、Converter、サイズ情報、オフセット情報などを生成します。

これによって、たとえば Python で動作する制御プログラムと、C++ で動作する物理シミュレータが、同じ PDU 定義を共有できます。

PDU Registry の役割は、単にデータ型を集めたライブラリを提供することではありません。一つの共通データ定義から、異なる言語や実行環境で利用可能なデータ表現を展開することで、箱庭エコシステム全体の共通データモデルを成立させます。

PDU の生成物には、用途に応じて次のようなものがあります。

- 言語別の型定義
- 固定オフセット形式の Converter
- オフセット情報
- PDU サイズ情報
- CDR 形式向け Converter やサイズ情報

これらは同じものではありません。たとえば、言語別の型定義が生成できることと、その言語で固定オフセット形式や CDR 形式の変換ができることは別の Capability です。

詳細な対応言語、生成物、Converter については、Catalog の `hakoniwa-pdu-registry` を参照してください。

## 6. MBody Registry - 物理ボディモデルを各実行環境へ展開する

箱庭では、通信データだけでなく、ロボットなどの物理的な「身体」を複数の実行環境で扱う必要があります。

この役割を担うのが MBody Registry です。

PDU Registry が「アセット間で交換するデータ」を共通化する仕組みだとすれば、MBody Registry は「シミュレーション空間に存在する物理ボディ」を共通化し、各実行環境へ展開するための仕組みと位置付けられます。

共通のモデル定義を起点として、URDF、MJCF、GLB など、各シミュレータや可視化環境が必要とする形式へ展開します。

これによって、一つのロボットモデルを、物理シミュレーション、3D 可視化、その他の実行環境で共有しやすくなります。

PDU Registry が「何を伝えるか」を共通化する仕組みだとすれば、MBody Registry は「何を動かすか、何を存在させるか」を共通化する仕組みと捉えることができます。

詳細な入力形式、生成可能なアセット、対応実行環境については、Catalog の `hakoniwa-mbody-registry` を参照してください。

## 7. PDU Endpoint - PDU 通信の出入口

PDU Endpoint は、箱庭アセットが PDU を送受信するための通信インタフェースを提供します。

PDU Endpoint の重要な役割は、アプリケーションと具体的な通信方式を分離することです。

箱庭アセットから見ると、PDU を読み書きするという基本的な操作を行います。その PDU が実際には、同一 PC 内の共有メモリにあるのか、TCP で別プロセスへ送られるのか、UDP で通信するのか、WebSocket でブラウザと接続されているのか、Zenoh などの通信基盤を利用するのか、といった違いを Endpoint の設定によって切り替えられます。

また、一つの Endpoint で複数の PDU を扱うことができます。

PDU のデータ構造は PDU 定義として記述し、Endpoint の接続方法は Endpoint 定義 JSON として記述します。これにより、

- どのようなデータを扱うか
- どのような方法で接続するか

を分離できます。

さらに PDU Endpoint は、用途に応じて PDU を扱うための Cache、Buffer、Queue などの仕組みも提供します。

PDU Endpoint は単なる通信ライブラリではなく、箱庭エコシステムにおける「PDU の共通出入口」として機能します。

詳細な通信方式、Endpoint 定義、データアクセス方式については、Catalog の `hakoniwa-pdu-endpoint` を参照してください。

## 8. PDU Bridge Core - PDU の転送を制御する

PDU Endpoint が「PDU の出入口」を提供するのに対し、PDU Bridge Core は Endpoint 間で PDU をどのように転送するかを制御します。

Bridge では、

- どの Endpoint から
- どの Endpoint へ
- どの PDU を
- どのタイミングで

転送するかを定義します。

代表的な転送ポリシーとして、Immediate、Throttle、Ticker などがあります。

たとえば、あるシミュレータが 100 Hz で PDU を更新していても、可視化アプリケーションには 10 Hz で転送したい、といった構成を Bridge の転送ポリシーとして表現できます。

Bridge の転送対象、経路、ポリシーは JSON で宣言的に定義します。

ここで重要なのは、箱庭が「通信」と「転送」を別の責務として扱っていることです。

- PDU Endpoint は「どう接続するか」を担当する
- PDU Bridge Core は「何を、どこへ、いつ転送するか」を担当する

この分離によって、通信プロトコルそのものを変更することなく、システム全体の PDU 転送構成や転送頻度を変更できます。

詳細な転送ポリシー、ルーティング、Bridge 設定については、Catalog の `hakoniwa-pdu-bridge-core` を参照してください。

## 9. PDU RPC - PDU による Request / Response 通信

PDU は、継続的な状態データの交換だけでなく、処理の要求と応答にも利用できます。

PDU RPC は、PDU 通信基盤の上に Request / Response 型の通信モデルを提供します。

通常の PDU 通信では、センサ値や位置情報などの状態を継続的に交換します。一方、RPC では、「この処理を実行してほしい」という Request を送り、その実行結果を Response として受け取ります。

これによって、箱庭の PDU 通信モデルを維持したまま、サービス呼び出しに近い通信を構成できます。

PDU RPC も、利用する PDU、サービス、接続構成などを JSON ベースの設定で宣言的に扱います。これにより、RPC の通信経路や接続関係をアプリケーションロジックから分離できます。

PDU RPC は、箱庭の上位サービスを構築するための基盤としても利用できます。たとえば Conductor Light では、PDU RPC を利用して時刻同期に必要な処理を提供します。

つまり PDU RPC は、PDU Endpoint などの通信基盤を利用し、その上に「処理呼び出し」という通信パターンを提供するレイヤとして位置付けられます。

詳細な RPC モデル、Request / Response 定義、利用方法については、Catalog の `hakoniwa-pdu-rpc` を参照してください。

## 10. シミュレーション時間を協調させる

複数のシミュレータやプログラムを組み合わせる場合、それぞれの時間進行をどのように扱うかが重要になります。

特に、複数のプロセスやコンピュータに分散すると、それぞれのシミュレーション時刻は暗黙には一致しません。

箱庭は、複数のアセットが協調してシミュレーションを進めるための時刻同期機構を提供します。

現在、システム構成や用途に応じて、主に以下の選択肢があります。

### Core PRO

一つの PC 上で、共有メモリを利用して複数の箱庭アセットを実行する構成で利用します。

ローカル環境で高速な PDU アクセスとシミュレーション時刻の協調が必要な場合に適しています。

詳細については、Catalog の `hakoniwa-core-pro` を参照してください。

### Conductor PRO

複数のコンピュータに箱庭アセットを分散配置するシミュレーション向けの商用コンポーネントです。

ノード間で箱庭時刻を協調させ、分散シミュレーションを構成するために利用します。

詳細については、Catalog の `hakoniwa-conductor-pro` を参照してください。

### Conductor Light

Conductor Light は、時刻同期機能を軽量な独立サービスとして利用するためのコンポーネントです。

共有メモリを前提とした箱庭ランタイムに時刻同期を組み込むのではなく、PDU RPC などの箱庭通信基盤を利用して時刻同期機能を提供します。

これにより、より疎結合な構成で時刻同期 Capability を利用できます。

Core PRO、Conductor PRO、Conductor Light は、いずれも「複数の実行主体の時間進行を協調させる」という共通の目的を持ちますが、想定するシステム構成と実現方式が異なります。

詳細については、Catalog の `hakoniwa-conductor-light` を参照してください。

## 11. 箱庭ベースエコシステムの全体像

ここまで説明したコンポーネントは、それぞれ独立した役割を持っています。

| コンポーネント | 主な役割 |
| --- | --- |
| PDU | 箱庭アセット間で交換するデータ |
| PDU Registry | PDU の共通定義から言語・実行環境別のデータ型と Converter を生成する |
| MBody Registry | 物理ボディの共通定義からシミュレータ・可視化環境向けのアセットを生成する |
| PDU Endpoint | PDU 通信の出入口を提供し、通信方式を抽象化する |
| PDU Bridge Core | Endpoint 間の PDU 転送、ルーティング、転送ポリシーを制御する |
| PDU RPC | PDU 通信上で Request / Response 型の処理呼び出しを実現する |
| Core PRO | 単一 PC・共有メモリ環境でシミュレーション実行と時刻同期を支える |
| Conductor PRO | 分散ノード間のシミュレーション時刻を協調させる |
| Conductor Light | 時刻同期 Capability を軽量な独立サービスとして提供する |

役割で見ると、箱庭のベースエコシステムは次のように捉えることができます。

### モデルを定義する

- PDU Registry
- MBody Registry

### データを通信する

- PDU Endpoint

### データを転送する

- PDU Bridge Core

### 処理を呼び出す

- PDU RPC

### 時間を協調させる

- Core PRO
- Conductor PRO
- Conductor Light

このベースエコシステムの上に、MuJoCo、Godot、Hakoniwa Drone、Athrill などのシミュレーションアセットや、ROS 2、Foxglove、Three.js などの外部システムとの連携コンポーネントを組み合わせることで、目的に応じたシミュレーションシステムを構成できます。

## 12. Ecosystem Guide、Runtime Primer、Catalog、Recipe の関係

本ドキュメントは、箱庭エコシステム全体の「地図」を提供することを目的としています。

各ドキュメントの役割は次のように分けます。

### Ecosystem Guide

箱庭の基本概念と、各コンポーネントの役割・関係性を理解するためのドキュメントです。

PDU とは何か、PDU Registry、MBody Registry、Endpoint、Bridge、RPC、Core / Conductor がエコシステムの中でどの位置にあるかは、本ドキュメントで説明します。

### Runtime Primer

箱庭ランタイムを実際に構成・実行するときに必要な前提知識を説明します。

たとえば、共有メモリランタイムの要件、`pdutypes` と `pdudef` の使い分け、アセットと外部クライアントの違い、ランチャー、起動順序、実行時検証などを扱います。

Runtime Primer は「箱庭とは何か」を説明するドキュメントではなく、「箱庭ランタイムをどう正しく動かすか」を説明するドキュメントです。

### Assets Guide

MuJoCo、Godot、Hakoniwa Drone、Athrill など、箱庭エコシステム上で利用可能な代表的なアセットと、その位置付けを理解するためのドキュメントです。

### Catalog

目的に必要な具体的なコンポーネント、その Capability、Interface、制約を調べるための機械可読な情報源です。

### Recipe

複数のコンポーネントを組み合わせ、具体的なシステム構成へ落とし込むための情報です。

### Demo

Recipe に基づいて実際に動作するシステムです。

全体としては、次の流れを想定しています。

```text
Ecosystem Guide
  -> Runtime Primer / Assets Guide
  -> Catalog
  -> Recipe
  -> Demo
```

この構造によって、人間だけでなく AI も、箱庭エコシステムの全体像を理解した上で、ユーザーが実現したいことに応じて必要なコンポーネントを選択し、組み合わせられることを目指します。
